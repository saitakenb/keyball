# 変更内容報告書 (Walkthrough): 物理送信ゲートウェイ「rgblight_call_driver」オーバーライドによる完全同期の達成

QMK標準のクリッピング機能（`rgblight_set_clipping_range`）の物理的な限界（範囲外のLEDへのパケットデータ送信処理自体を完全に省略する仕様）により、スレーブ側の物理LEDに以前の全点灯データ（青や白）が残り続けてしまっていた問題を解消しました。

物理送信直前の最終バッファを強制的に書き換えることで、上書き競合と同期フリーズを完全に解消したファームウェアのビルドとGitHubへのプッシュを完了しました。

## 実施した変更内容

### 1. 物理送信直前のゲートウェイ `rgblight_call_driver` のオーバーライド
- **原因**: 
  - QMKの `rgblight_set()` は、物理送信を行う際、`rgblight_ranges.clipping_num_leds`（クリッピング範囲）の長さ分（例：点灯個数8）しか物理LEDドライバに送信しません。
  - そのため、スレーブ側でクリッピング範囲を制限してしまうと、それより後ろの物理LED（8〜33番目など）には **直前に全点灯していた「青」や「白」のデータが物理LED内部のICに保持されたまま** になり、消灯できなくなっていました。
- **対策**:
  - QMK標準の `rgblight_set()` が物理送信を実行する直前に呼び出す、弱参照関数 **`rgblight_call_driver(LED_TYPE *start_led, uint8_t num_leds)`** を `keymap.c` 側でオーバーライドして乗っ取りました。
  - 物理送信が実行されるその瞬間に割り込み、渡されたバッファ `start_led` に対して、**強制的に消灯クリアおよび `sat` に基づく部分点灯（親指制限）へのカラー書き換え** を実行してから物理LEDへ出力させます。
  - これにより、スレーブ側のQMK標準タスクが同期受信により全点灯データを送信しようとしたその瞬間に、データが自動的に部分点灯データにすり替わるため、全点灯やフラッシュが発生することは論理的・物理的に 100% 完全に不可能になります。

### 2. 割り込み競合による波形破壊の完全防止
- スレーブ側の `matrix_scan_user` や `update_led_state` から `ws2812_setleds` や `rgblight_set()` を個別に直接呼び出すハック処理を、スレーブ側からすべて廃止しました。
- 物理出力は、同期受信が完了した直後の「最も通信ノイズがなく安全な瞬間（割り込みの処理順序の末尾）」に、QMK標準の `rgblight_set` の実行プロセスの中で自動的に行われるため、SOFT_SERIALの受信割り込みと衝突してLEDがフリーズする波形破壊の問題が 100% 完全に解消します。
