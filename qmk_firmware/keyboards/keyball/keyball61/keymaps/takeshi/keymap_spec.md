# Keyball61 "takeshi" キーマップ 仕様書

> 対象ファームウェア: QMK Firmware 0.22.14
> 対象キーボード: Keyball61（左右分割型、右手側トラックボール）
> ファームウェアサイズ: 28,242 / 28,672 バイト（残り430バイト）

## 1. ハードウェア構成

| 項目 | 詳細 |
|---|---|
| キーボード | Keyball61（左右分割スプリット型） |
| トラックボール | 右手側に搭載 |
| マイコン | Pro Micro（AVR ATmega32U4） |
| 通信 | USB（メイン側）+ QMK シリアルスプリット同期（サブ側） |
| OLED | 両側に搭載 |
| RGB LED | 左手側：37個 / 右手側：34個 |

### LED 物理配置

| 側 | 裏面親指（優先点灯） | 前面 | 合計 |
|---|---|---|---|
| **左手** | インデックス 29〜36（8個） | インデックス 0〜28（29個） | **37個** |
| **右手** | インデックス 0〜6（7個） | インデックス 7〜33（27個） | **34個** |

## 2. レイヤー構成

| レイヤー | 名称 | 遷移方法 | LED色 |
|---|---|---|---|
| **Layer 0** | デフォルト（英字） | 起動時 / `TO(0)` | 白 |
| **Layer 1** | 数字・記号・カーソル | `LT(1,...)` / `TO(1)` | 薄紫（ラベンダー） |
| **Layer 2** | ファンクション・マウス | `LT(2,...)` / `TO(2)` | シアン（水色） |
| **Layer 3** | RGB・CPI・メディア調整 | `LT(3,...)` / `OSL(3)` | 青 |

> Layer 3 では **スクロールモードが自動的にON** になります（`keyball_set_scroll_mode(true)`）。

## 5. RGB LED 仕様

### 5-1. 点灯パターン仕様

起動時は **裏面親指LED8個（両側とも）** が点灯した状態になります。

**左手側の点灯順序**（点灯数を増やした時）:
1. まず裏面親指LED（インデックス 29→36 の順、最大8個）
2. 次に前面LED（インデックス 0→28 の順、最大29個）

**右手側の点灯順序**（点灯数を増やした時）:
1. まず裏面親指LED（インデックス 0→6 の順、最大7個）
2. 次に前面LED（インデックス 7→33 の順、最大27個）

### 5-2. レイヤー連動カラー

| レイヤー | 色 | Hue | Sat（実際の彩度） |
|---|---|---|---|
| Layer 0 | 白 | 170 | 0 |
| Layer 1 | 薄紫（ラベンダー） | 200 | 160 |
| Layer 2 | シアン（水色） | 128 | 255 |
| Layer 3 | 青 | 170 | 255 |

> 色の彩度は **各デバイスがレイヤー状態から自律計算** するため、同期コストが最小化されています。

### 5-3. 点灯数の調整

| 操作 | 効果 |
|---|---|
| Layer 3 の `RGB_MOD` を押す | 点灯数を +1（最大37） |
| Layer 3 の `RGB_RMOD` を押す | 点灯数を -1（最小0） |
| OLED表示 | ` L:08` のように現在の点灯数を表示 |

### 5-4. 全体の輝度（明るさ）の調整

| 操作 | 効果 |
|---|---|
| Layer 3 の `RGB_VAI` を押す | 全体の明るさを増加 |
| Layer 3 の `RGB_VAD` を押す | 全体の明るさを減少 |
| 初期輝度 | 50%（Val=127） |

> 輝度はスレーブ側にも自動同期されます。

### 5-5. 内部実装（QMK変数マッピング）

QMK標準の同期パケットを活用しつつ、スプリット環境での競合を完全回避するため、変数を再定義しています：

| QMK変数 | 本来の意味 | このファームウェアでの役割 |
|---|---|---|
| `rgblight_config.hue` | 色相 | レイヤーの色相（同期で転送） |
| `rgblight_config.sat` | 彩度 | **点灯LED個数（0〜37）** |
| `rgblight_config.val` | 明度 | **全体の明るさ（輝度）** |
| `rgblight_config.mode` | エフェクトモード | `1`（静的点灯）に固定 |

**スレーブ側全点灯防止のための周期上書き**:
スキャン50回（約30〜40ms）に1回、両側のデバイスで `update_led_state()` を強制実行し、QMK標準タスクによる全点灯の上書きをリアルタイムで打ち消します。

## 7. ビルド設定（rules.mk / config.h）

| 設定 | 値 | 理由 |
|---|---|---|
| `CAPS_WORD_ENABLE` | `no` | フラッシュサイズ削減のため無効化 |
| `RGBLIGHT_ANIMATIONS` | `no` | 同上、かつ最大モード数を1に制限することで意図しない動作を防止 |
| `BOTH_SHIFTS_TURNS_ON_CAPS_WORD` | 無効（コメントアウト） | Caps Word無効化に伴い不要なため |

## 8. 左右接続判定の仕様

QMKのタイマーAPI（`timer_read()`）はSPI通信（トラックボール）の割り込み処理と競合してフリーズする可能性があるため、**スキャンループのカウンター方式** を採用しています。

| 条件 | 判定結果 |
|---|---|
| トラックボールが検出された（`keyball.this_have_ball == true`） | 右手（サブ）と確定 |
| スキャンが5000回実行されてもトラックボールが検出されない | 左手（マスター）と確定 |

> 一度確定した判定は以降変更されません（static latching）。

---

## 9. 対応する接続パターン

| 接続方法 | マスター | スレーブ |
|---|---|---|
| **左メイン** | 左手側（USB接続） | 右手側 |
| **右メイン** | 右手側（USB接続） | 左手側 |

両パターンとも、LEDの点灯数制限・レイヤー色連動・輝度調整の全機能が両側で完全に動作します。
