# Keyball61 / QMK スプリット環境開発におけるデバッグ知見集 (Debugging Knowledge Base)

本ドキュメントは、Keyball61（左右分割型自作キーボード）のカスタムファームウェア開発において発生した一連のバグ、その技術的な根本原因、および解決策を整理した永続的な知見集です。今後のAIコードアシスタントおよび開発者がこれを参照し、バグの再発やデグレードを完全に防止することを目的としています。

---

## 1. 発生したバグの歴史と根本原因の分析

### バグ①: 左右LED의 同期不良・フリーズ
* **現象**: 左右のLEDの色や状態が同期せず、しばらく操作するとフリーズする。
* **原因**: Keyballの共通コード（`keyball61.c`）がレイヤー切り替え時に自動実行する `rgblight_set_clipping_range()` と、独自実装のクリッピング制御が衝突し、QMK内部のLED配列範囲が初期化されていたため。
* **対策**: クリッピング範囲の動的変更を廃止し、`led` 配列メモリを直接操作して消灯（`(0,0,0)`）を書き込むロジックに移行した。

### バグ②: LED点灯個数の増減トグルが機能しない
* **現象**: 点灯数を変更するキーを押しても、LEDの点灯個数が一切変化しない。
* **原因**: QMKのサイズ削減のためアニメーション効果を無効化（`RGBLIGHT_ANIMATIONS = no`）した結果、QMKの最大エフェクトモード数が `1` に制限されていた。そのため、QMKの `rgblight_mode_noeeprom()` に `2` 以上の値（点灯数）を設定しようとしても、境界チェックによってすべて拒否されていた。
* **対策**: 境界チェックのない `rgblight_config.hue` (色相値) を「点灯個数（0〜37）」として流用して同期させる方法を試みた。

### バグ③: 右メイン接続時に左右とも赤で全点灯しフリーズする
* **現象**: 右手側にUSBを挿した際、左右とも赤色で全点灯したままキー入力やレイヤー切り替えに一切反応しなくなる。
* **原因**: トラックボールセンサー（SPI通信）の初期化や通信処理の際、マイコン内部で割り込み禁止が多発するため、QMK標準のミリ秒タイマー（`timer_read()`）がフリーズし、左右判定のタイムアウト判定が無限ループ（ハングアップ）を起こし、右手側が自身を左手と誤認していた。
* **対策**: タイマーAPIへの依存を完全に排除し、スキャンループの実行回数カウンタ（5000回スキャン＝約2〜3秒相当）による決定論的なラッチ判定に変更した。

### バグ④: 右メイン接続時、スレーブ（左手側）が赤で全点灯のまま反応しない
* **現象**: 右メイン接続時、右手側（マスター）は動くが、スレーブ（左手側）が赤で全点灯のまま固定される。
* **原因**: スレーブ側にはキーイベントがないため、`matrix_scan_user` 内の「キー入力時の状態変化検知（`last_state` 比較）」が走らず、QMKの同期パケット受信時にQMKの標準タスク（`rgblight_task`）が裏でデフォルトの赤色を上書きし続けていた。
* **対策**: 状態変化検知を廃止し、スキャン100回に1回、無条件で `update_led_state()` を強制実行してスレーブ側の赤色上書きを時間的に圧倒するロジック（周期強制上書き）を導入した。

### バグ⑤: スレーブ側が常に赤で全点灯する
* **現象**: 左右の接続を入れ替えても、スレーブ側デバイスが常に赤で全点灯する。
* **原因**: 点灯数を `rgblight_config.hue` (色相) にマッピングしたため、スレーブ側が同期で受信するHue値（例: `8`）はHSVカラー変換において **「赤色（Red）」** を意味していた。そのため、スレーブ側がシリアル通信を受信した瞬間に、QMK標準タスクが自動的に「受信した色（Hue=8＝赤色）で全LEDを塗りつぶして出力」してしまっていた。
* **対策**: `hue`/`sat` に「正しいレイヤーの色」をマッピングし、点灯個数は `val` (明度) にマッピングする設計に変更した。

### バグ⑥: 明るさ（輝度）調整機能の喪失
* **現象**: 点灯数を `val` (明度) にマッピングしたため、LEDカラー生成時に輝度を255固定せざるを得なくなり、`RGB_VAI`/`RGB_VAD` による全体の明るさ調整が機能しなくなった。
* **対策**: `val` を明るさ調整用として残し、点灯個数を `mode` (モード変数) へ直接代入して同期させる設計に変更した。

### バグ⑦: スレーブ側が全点灯のまま個数制限が効かない（modeリセット）
* **現象**: 起動時は動くが、スレーブ側が「全点灯」のまま個数制限が機能しない。
* **原因**: マスターからスレーブへ `mode`（値＝個数）が同期された瞬間、スレーブ側のマイコンでもQMKの受信処理の中で `mode` の境界チェック（最大1）が走り、強制的に **`1`**（静的点灯モード＝全点灯）にリセットされてしまっていた。
* **対策**: 境界チェックがなく自動同期される **`rgblight_config.sat` (彩度)** を「点灯個数（0〜37）」にマッピングする設計に変更した。

### バグ⑧: 点灯数を調整した「後」にスレーブ側の色が変わらなくなる（同期の完全遮断）
* **現象**: 最初は左右ともレイヤーに反応するが、点灯数キー（`RGB_MOD` / `RGB_RMOD`）を押して点灯数を変えた瞬間に、スレーブ側の色が変わらなくなる。
* **原因**: 
  - 全点灯フラッシュを防ぐためにクリッピング範囲を `(0, 0)` や `(0, 1)` に制限した状態で、QMK標準の `rgblight_sethsv_noeeprom()` を呼び出すと、QMKの内部処理で描画対象がない（または極小）と判断され、スプリット同期パケットの送信処理が完全にスキップされていました。
  - スレーブ側も、クリッピング範囲が `(0, 1)` に制限されたままになっていたため、同期受信時に親指以外のLEDをクリア・更新することができず、永続的に全点灯し続けていました。
* **対策**: 
  - QMKの描画処理が全LEDを塗りつぶしてしまう `rgblight_sethsv_noeeprom()` を一切呼び出さず、`rgblight_config` に値を代入後に、QMK公式の同期ビットフラグ **`rgblight_status.change_flags |= RGBLIGHT_STATUS_CHANGE_HSVS;`** を直接セットして同期をトリガーするように修正。これにより、クリッピング制限をかけずにフラッシュを完全回避しつつ、同期パケットを100%確実に送信可能にしました。
  - スレーブ側は、後述の「常時 `(0, 0)` 防御」により、標準タスクの上書きを防ぎます。

### バグ⑨: 全71個バッファ一括操作によるLEDのちらつき（データ競合）
* **現象**: 左手側の親指LED（インデックス29）などが、高頻度でパタパタと点滅（チラツキ）する。
* **原因**: 左右合計の全71個のLEDバッファをマスター側で無理やり描画しようとしたため、QMKのスプリット同期受信処理による上書きデータと、ローカルの描画処理がLED配列上で激しく競合（衝突）していたため。
* **対策**: マスター側もスレーブ側も、自分自身のローカル物理LED（L=37, R=34）のみを `update_led_state()` で自律描画する設計に戻し、LED配列上のデータ衝突（チラツキ）を100%完全に解消しました。

### バグ⑩: スレーブ側のQMK標準タスクによる全点灯自動上書き
* **現象**: スレーブ側のLEDが、一時的に個数制限されたとしても、即座にすべてのLEDが点灯する（全点灯に戻る）。
* **原因**: スレーブ側デバイスは、マスターから同期データを受信した際、QMKのバックグラウンド標準タスク（`rgblight_task()`）が自動的にループ起動し、受信したHSV設定値に基づいて常に全LEDをその色（全点灯状態）で塗りつぶして出力し続けていたため。
* **対策**: スレーブ側デバイスのエフェクト・クリッピング範囲を **常時 `(0, 0)`（描画LED数0）** に設定しておくことで、標準タスクの塗りつぶし・出力を完全に無効化（シャットアウト）しました。`update_led_state()` の描画の瞬間だけ一時的に最大数に戻して `rgblight_set()` を実行し、直後に即座に `(0, 0)` に戻します。

## 2. 再発防止のための開発設計原則（黄金律）

今後、このリポジトリのRGB LEDやスプリット同期のコードを変更する際は、以下の原則を厳格に遵守してください。

### 原則1: 左右判定には必ず `is_keyboard_left()` を使用すること
- QMK標準の左右判定APIである **`is_keyboard_left()`** は、スプリットの通信初期化シーケンスの中でEEPROMやハードウェアピンの状態を読み取って確定されるため、スレーブ側であっても 100% 正確に動作します。
- タイマーラッチやトラックボールの有無（`keyball.this_have_ball`）による判定は、スレーブ側での初期化順序の違いにより誤判定を引き起こし、左右の描画ロジックがすり替わる原因になるため、使用してはなりません。

### 原則2: アニメーション無効時は `rgblight_config.sat` を点灯数にマッピングすること
- 境界チェックや自動リセットが走る変数（`mode`）や、全体の明るさ調整に必要な変数（`val`）を個数同期に流用してはいけません。以下のマッピングを維持してください：
  - **`hue`**: レイヤー別の色相（同期用）
  - **`sat`**: LED点灯個数 (0〜37)（境界チェックがなく安全に同期可能）
  - **`val`**: 全体の輝度（明るさ調整機能を完全に維持）
  - **`mode`**: `1` (静的点灯モード) に固定
- レイヤーごとの本来の彩度（白=0, 紫=160, 他=255）は、スレーブ側およびマスター側が「現在のレイヤー状態」からローカルで自律決定して描画色を生成してください。

### 原則3: 同期は `change_flags` を直接セットしてトリガーすること
- レイヤー切り替えや点灯数調整による変更をスレーブに送る際、全点灯描画を引き起こす `rgblight_sethsv_noeeprom()` を呼び出してはいけません。
- `rgblight_config` の構造体の値を直接書き換えた後に、**`rgblight_status.change_flags |= RGBLIGHT_STATUS_CHANGE_HSVS;`** を実行して同期をトリガーしてください。これにより、全点灯フラッシュは1ナノ秒も発生せず、トランスポート同期パケットも確実に送信されます。

### 原則4: スレーブ側は常時エフェクト範囲 `(0, 0)` で自動更新を防御すること
- スレーブ側でQMK標準タスクが自動上書きを実行するのを防ぐため、スレーブデバイスはクリッピングおよびエフェクト範囲を **常時 `(0, 0)` に設定** してください。
- `update_led_state()` の描画の瞬間だけ一時的に最大数に戻して `rgblight_set()` を実行し、直後に即座に `(0, 0)` に戻すことで、標準タスクの塗りつぶしを完全に無効化（シャットアウト）します（マスター側は同期送信のため最大数のままにしておく）。
